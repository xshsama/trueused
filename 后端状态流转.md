后端逻辑梳理与 Mock 数据演练 (寄售模式)

本文档通过模拟一次完整的 iPhone 13 交易，展示后端数据库在不同阶段的状态流转 (Status) 和关键数据变化。

🟢 第一阶段：寄售入库流 (Seller -> Platform)

核心目标：将卖家手里的“闲置物品”转化为平台仓库里的“标准库存”。

步骤 1：卖家发起寄售申请

场景：卖家 Mike 在 App 上点击“发布”，填写信息并提交。

API: POST /api/consignments

后端逻辑：创建寄售单，状态初始化为 CREATED。

Mock Database (Consignment Table):

{
"id": "C_20241025_001",
"seller_id": "U_MIKE",
"title": "iPhone 13 128G 国行",
"status": "CREATED", // <--- 初始状态
"shipping_method": "PICKUP", // 预约顺丰上门
"product_id": null // 此时还未生成正式商品
}

步骤 2：物流揽收 (卖家发货)

场景：顺丰快递员上门取件成功，物流 API 回调平台。

API: WEBHOOK /logistics/callback

后端逻辑：更新寄售单状态，绑定物流单号。

Mock Database Update:

{
"id": "C_20241025_001",
"status": "SHIPPED", // <--- 变更为运输中
"tracking_no_inbound": "SF1000000001", // 入仓物流单号
"shipped_at": "2024-10-25 14:00:00"
}

步骤 3：仓库签收 & 触发验货

场景：包裹到达上海仓，操作员扫码签收。

API: POST /api/warehouse/receive

后端逻辑：

状态变更为 RECEIVED。

立即创建验货任务 (Inspection Task)。

Mock Database Update:

// 寄售单表
{
"id": "C_20241025_001",
"status": "RECEIVED"
}

// 新增：验货任务表 (Inspections)
{
"id": "INSP_001",
"consignment_id": "C_20241025_001",
"status": "PENDING", // 待验货
"product_snapshot": { ... }
}

步骤 4：验货完成 & 自动上架 (关键节点)

场景：验货引擎完成检测，判定为 S 级，生成报告。

后端逻辑 (自动化)：

验货任务结束，写入报告数据。

核心动作：在 Products 表创建一条正式商品记录。

寄售单状态完结。

Mock Database Update:

// 1. 验货任务表
{
"id": "INSP_001",
"status": "FINISHED",
"grade": "S",
"report_url": "[https://trueused.com/report/R001.pdf](https://trueused.com/report/R001.pdf)"
}

// 2. 新增：商品表 (Products) -> 正式库存
{
"id": "P_10001",
"consignment_id": "C_20241025_001",
"title": "iPhone 13 128G 国行 [S 级]",
"price": 3599.00,
"status": "ON_SALE", // <--- 上架出售中！
"stock": 1,
"warehouse_location": "SH_A_01_05" // 货架号
}

// 3. 寄售单表
{
"id": "C_20241025_001",
"status": "WAREHOUSED", // 已入库
"product_id": "P_10001" // 关联正式商品
}

🔵 第二阶段：交易出库流 (Buyer -> Platform)

核心目标：消耗库存，处理资金，完成履约。

步骤 5：买家下单 (锁库存)

场景：买家 Alice 看到商品，点击“立即购买”。

API: POST /api/orders

后端逻辑：

创建订单，状态 PENDING_PAYMENT。

锁库存：将商品状态改为 LOCKED，防止其他人购买。

Mock Database Update:

// 订单表 (Orders)
{
"id": "ORD_20241026_888",
"buyer_id": "U_ALICE",
"product_id": "P_10001",
"amount": 3599.00,
"status": "PENDING_PAYMENT", // 待付款
"expired_at": "2024-10-26 10:15:00" // 15 分钟倒计时
}

// 商品表 (Products)
{
"id": "P_10001",
"status": "LOCKED" // <--- 锁定中
}

步骤 6：支付成功 & 推送发货

场景：Alice 支付成功。

API: WEBHOOK /payment/alipay/notify

后端逻辑：

订单状态 -> PAID。

商品状态 -> SOLD_OUT。

自动推单：调用 WMS 接口，通知仓库发货。

Mock Database Update:

// 订单表
{
"id": "ORD_20241026_888",
"status": "PAID", // 已付款，等待仓库响应
"pay_time": "2024-10-26 10:05:00"
}

// 商品表
{
"id": "P_10001",
"status": "SOLD_OUT" // <--- 已售出
}

步骤 7：仓库发货 (出库)

场景：仓库打包完毕，顺丰揽收。

API: POST /api/internal/warehouse/ship

后端逻辑：

更新订单物流信息。

状态 -> SHIPPED。

Mock Database Update:

{
"id": "ORD_20241026_888",
"status": "SHIPPED", // 已发货
"tracking_no_outbound": "SF2000000002" // 发货物流单号
}

步骤 8：确认收货 & 资金结算

场景：Alice 收到货，点击“确认收货”。

API: POST /api/orders/{id}/confirm

后端逻辑：

订单 -> COMPLETED。

分账：

平台收入：¥3599 \* 5% = ¥179.95

卖家(Mike)收入：¥3599 - ¥179.95 = ¥3419.05

将钱打入 Mike 的钱包余额。

Mock Database Update:

// 订单表
{
"id": "ORD_20241026_888",
"status": "COMPLETED",
"finish_time": "2024-10-28 18:00:00"
}

// 卖家钱包表 (Wallets)
{
"user_id": "U_MIKE",
"balance": 3419.05, // <--- 余额增加
"history": [
{ "type": "INCOME", "amount": 3419.05, "source": "ORD_20241026_888" }
]
}

🔴 异常分支 Mock (如果是坏得呢？)

分支场景：验货不通过

如果在 步骤 4 中，验货发现是假货。

Mock Update:

// 验货任务
{ "id": "INSP_001", "status": "REJECTED", "grade": "FAIL" }

// 寄售单
{
"id": "C_20241025_001",
"status": "REJECTED" // <--- 验货驳回
}

// 生成退货单 (ReturnOrder)
{
"consignment_id": "C_20241025_001",
"tracking_no": "SF3000000003", // 退回给卖家的单号
"reason": "COUNTERFEIT" // 假货
}

此时不会生成 Product 记录，商品永远不会上架。
